SCROLL &amp; ANIMATION TECHNIQUES
IntersectionObserver (no library required)
The correct approach for a portfolio that must score 95+ on Lighthouse. No Framer Motion, no AOS.js — just native browser APIs.
Pattern:
const observer = new IntersectionObserver((entries) =&gt; { entries.forEach(e =&gt; { if (e.isIntersecting) e.target.classList.add(&apos;visible&apos;); }); }, { threshold: 0.15 });

CSS:
.reveal { opacity: 0; transform: translateY(28px); transition: opacity 0.55s ease-out, transform 0.55s ease-out; } .reveal.visible { opacity: 1; transform: translateY(0); }

Stagger:
.reveal:nth-child(1) { transition-delay: 0ms; } .reveal:nth-child(2) { transition-delay: 80ms; } .reveal:nth-child(3) { transition-delay: 160ms; }

Lerp Cursor (pure JavaScript)
let mouseX = 0, mouseY = 0, cursorX = 0, cursorY = 0;
document.addEventListener(&apos;mousemove&apos;, e =&gt; { mouseX = e.clientX; mouseY = e.clientY; });
function animate() {
  cursorX += (mouseX - cursorX) * 0.15; // 0.15 = lerp factor
  cursorY += (mouseY - cursorY) * 0.15;
  cursor.style.transform = `translate(${cursorX}px, ${cursorY}px)`;
  requestAnimationFrame(animate);
}
animate();

CORE WEB VITALS — 2025/2026 STANDARDS
CRITICAL UPDATE: FID (First Input Delay) was replaced by INP (Interaction to Next Paint) in March 2024. If you&apos;re reading content from before 2024, ignore FID references.

LCP (Largest Contentful Paint): ≤ 2.5s = Good · ≤ 1.5s = Exceptional. Optimize: preload hints, image compression, SSR.
INP (Interaction to Next Paint): ≤ 200ms = Good. Measures ALL interactions, not just first. Far stricter than FID.
CLS (Cumulative Layout Shift): ≤ 0.1 = Good · ≤ 0.05 = Exceptional. Reserve sizes for all images, avoid injecting above fold.

Target for your portfolio: LCP &lt; 1.5s · INP &lt; 100ms · CLS &lt; 0.05 · Lighthouse Performance ≥ 95 · Accessibility 100 · SEO 100

PART 10
WRITING RULES &amp; COPY GUIDE
The quality of your writing IS a signal of the quality of your thinking. Every word on your portfolio is being evaluated — especially by non-technical reviewers and scholarship committees.

THE DUAL-REGISTER RULE (Most Important)
Every project description must work in TWO registers simultaneously.

Human register (LARGE, top line): The problem you solved and for whom — no jargon
Technical register (SMALL, below): The stack, metrics, and architecture

Non-tech reader sees the first line and understands. Tech reviewer reads both. Scholarship committee reads both and sees research maturity.

WEAK: &quot;Implemented RESTful API with JWT authentication and TF-IDF vectorization for sentiment classification.&quot;

STRONG: &quot;Built a tool that helps healthcare workers identify at-risk pregnancies using an AI model — available as a free public API. [SMALL: Python · REST · Swagger UI · Flutter · ML model accuracy: 88%]&quot;

PHRASES TO AVOID
DON&apos;T say this ❌
SAY this instead ✅
Passionate about technology
I care about [specific thing] because [specific reason]
I love solving problems
Built X for Y people, which reduced Z by W%
Detail-oriented team player
Led a team of X that delivered Y in Z weeks
Proficient in React
React: used in 5 production projects, most recently [name]
Responsible for...
Delivered / Built / Launched / Reduced / Grew...
I am interested in...
I am currently building / studying / researching...
Experience with...
Deployed / Shipped / Published / Measured...

VOICE GUIDELINES
First person, present tense, active voice: &apos;I build&apos; not &apos;I have experience building&apos;
Confident, not arrogant: state facts, not claims (&apos;I shipped X&apos; not &apos;I am expert at X&apos;)
Short sentences in hero and about sections. Longer sentences acceptable in case studies.
No walls of text anywhere — max 3 sentences per paragraph
Grammar and punctuation matter enormously for scholarship committees — proofread everything
British or American English consistently — choose one, do not mix
